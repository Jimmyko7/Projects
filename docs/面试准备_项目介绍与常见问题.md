## 2. 如何介绍你的项目

### 一句话总结（电梯演讲）

> "我用 Python 从零构建了一个**完整的数据→AI 流水线**：从 TMDB 网站爬取 300 部电影数据，进行多维度数据分析和可视化，然后基于 LangChain + Chroma 搭建了 **RAG 检索增强生成的电影智能问答系统**，支持自然语言搜索和流式对话。此外还完成了 AI 聊天机器人和面向对象的教务管理系统。"

```
R (成果):    可运行的 RAG 问答系统，支持 Web UI 和 CLI 双模式
```

**面试中可展开的点**:

## 3. 面试高频问题

### Q1: "RAG 检索部分是怎么做的？"

**建议回答**：

1. **嵌入层**：阿里云 DashScope `text-embedding-v4`，1024 维向量
2. **分块策略**：`RecursiveCharacterTextSplitter`，chunk_size=500，overlap=50，按中文标点递归分割
3. **向量库**：Chroma 本地持久化，collection 名 `tmdb_movies`
4. **检索逻辑**（重点）：
   - 先判断查询意图（是否包含"最/排名/Top"等关键词）
   - 普通查询：`similarity_search(query, k=5)`
   - 排名/极值查询：扩大 k=10，并从 ChromaDB 全量拉取按数值字段排序
   - **关键优化**：语义检索基于向量相似度，不保证数值极值在前几名

### Q2: "为什么向量检索找不到评分最高的电影？"

**这是你实际踩过的坑，面试中讲出来是加分项**：

> "我在测试时发现问'评分最高的电影'，Chroma 语义搜索返回的结果评分都是 0 或者不是真正的最高分。原因是**向量相似度衡量的是语义相关性，不是数值大小**。两个电影在语义空间接近是因为类型/风格相似，而不是评分接近。
>
> 我的解决方案是：检测到极值类查询时，直接从 ChromaDB 拉取全量 metadata，按评分字段做数值排序，取 top-N。这样就绕过了语义检索在这个场景下的盲区。"

### Q3: "项目里用了哪些设计模式？"

| 项目 | 模式 | 体现 |
|------|------|------|
| P02 | 配置分离 (Config Object) | `config.py` 集中管理 URL/XPath/超时 |
| P02 | 策略模式 | `Retry` 重试策略 |
| P04 | 分层架构 | `StuGrade`(Model) + `EduManagement`(Service) |
| P05 | 模板方法 | `rag_engine.py` 统一检索入口 |
| P05 | Repository 模式 | Chroma 作为数据访问层 |

1. **容器化**: Dockerfile + docker-compose
2. **API 化**: FastAPI 封装 RAG 检索为 REST API
3. **向量库升级**: Chroma → Milvus/Qdrant（分布式、高并发）
4. **监控**: 检索延迟、LLM 调用成功率、Token 消耗
5. **CI/CD**: GitHub Actions（lint → test → build → deploy）
6. **测试**: 单元测试 + RAGAS 检索质量评估
7. **安全**: API Key 用 Secret Manager，不写环境变量

## 4. 技术深度问题（RAG 专项）

### Q6: "RAG 的核心流程是什么？"

```
用户问题 → Embedding → 向量相似度检索 (Top-K)
    → 格式化检索结果为文本上下文
    → 注入 LLM 提示词 (System Prompt + Context)
    → LLM 生成答案（基于上下文，减少幻觉）
```

### Q7: "Chunk Size 选 500 是为什么？"

- 太小（<200）：语义信息不完整，检索碎片化
- 太大（>1000）：噪音多，超出嵌入模型上下文窗口
- 500 是经验值，配合 overlap=50 保证边界信息不丢失
- 最佳实践：做 ablation study，对比不同 chunk size 的检索命中率

### Q8: "为什么用 Chroma 而不是 FAISS 或 Pinecone？"

| 方案 | 优势 | 劣势 |
|------|------|------|
| Chroma | 开箱即用、Python 原生、本地持久化 | 性能一般、不支持分布式 |
| FAISS | 速度极快、支持 GPU | 无持久化、需手动序列化 |
| Pinecone | 全托管、高可用 | 付费、数据出海 |
| Milvus | 分布式、高性能 | 部署复杂、资源占用大 |

> "对于 300 部电影、几百个向量的规模，Chroma 完全够用。如果到百万级，会考虑 Milvus。"

### Q9: "Embedding 模型为什么选 DashScope 而不是 OpenAI？"

1. **中文优化**: `text-embedding-v4` 在中英文混合场景优于 `text-embedding-ada-002`
2. **成本**: 阿里云模型更便宜
3. **可访问性**: 不需要海外 API 代理
4. **维度**: 1024 维，在精度和存储间平衡

### Q10: "如何评估 RAG 系统的检索质量？"

RAGAS 框架的 4 个核心指标：
- **Context Precision**: 检索到的文档中有多少是相关的
- **Context Recall**: 相关文档中有多少被检索到
- **Faithfulness**: 答案是否忠实于检索到的上下文
- **Answer Relevance**: 答案是否直接回应了问题
